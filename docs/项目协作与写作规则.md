# 项目协作与写作规则

## 1. 分支规则

项目使用三类分支：

`main dev feature/xxx`

### main 分支

main 是稳定分支，只保存可以运行、可以演示、可以部署的版本。

禁止直接在 main 上开发。

只有在以下情况才合并到 main：

`1. 一个阶段功能完成 2. Docker 部署验证通过 3. 冒烟测试通过 4. 负责人确认没有明显问题`

### dev 分支

dev 是日常集成分支。

所有功能开发完成后，先合并到 dev，确认没问题后再由负责人合并到 main。

### feature 分支

每个人开发功能时，都从 dev 拉自己的功能分支：

`git checkout dev git pull origin dev git checkout -b feature/功能名`

示例：

`feature/frontend-login feature/material-upload feature/question-answer feature/admin-review feature/docs-deploy`

---

## 2. 开发流程

### 开始开发前

`git checkout dev git pull origin dev git checkout -b feature/你的功能名`

### 开发过程中

每完成一个相对完整的小功能就提交一次，不要攒到最后一次性提交。

`git status git add -A git commit -m "feat: 实现资料发布表单"`

### 开发完成后

`git push origin feature/你的功能名`

然后在 GitHub 上发 Pull Request：

`feature/你的功能名 -> dev`

由负责人审核后合并。

---

## 3. 提交信息规则

提交信息格式：

`类型: 具体说明`

常用类型：

| 类型       | 说明          |
| -------- | ----------- |
| feat     | 新增功能        |
| fix      | 修复问题        |
| docs     | 修改文档        |
| style    | 调整格式，不影响功能  |
| refactor | 重构代码        |
| test     | 添加或修改测试     |
| chore    | 构建、配置、依赖等杂项 |

示例：

`feat: 实现用户登录接口 feat: 实现资料发布页面 fix: 修复 Docker 端口冲突 fix: 修复中文乱码问题 docs: 补充 Docker 部署说明 test: 增加资料模块接口测试 refactor: 调整用户模块 Service 分层 chore: 更新前端依赖配置`

不推荐：

`update 修改 111 改了一点 最终版`

---

## 4. Pull Request 规则

每个 PR 应该说明：

`1. 做了什么 2. 改了哪些模块 3. 如何验证 4. 是否有遗留问题`

PR 描述模板：

`## 本次改动 - ## 涉及模块 - ## 验证方式 - [ ] 本地启动后端成功 - [ ] 本地启动前端成功 - [ ] Docker 部署通过 - [ ] 冒烟测试通过 - [ ] 浏览器页面验证通过 ## 备注 -`

如果只是文档改动，可以写：

`## 本次改动 - 补充 Docker 部署教程 - 补充开发模式说明 ## 验证方式 - [x] 检查文档链接可读`

---

## 5. 代码写作规则

### 后端代码

后端遵循分层结构：

`Controller -> Service -> Mapper -> Database`

要求：

`Controller 只处理请求参数和响应，不写复杂业务逻辑 Service 写业务规则 Mapper 只负责数据库访问 VO 返回给前端 DTO 接收前端请求 Entity 对应数据库表`

不要在 Controller 里直接写 SQL。

不要直接把 Entity 暴露给前端，优先使用 VO。

统一返回格式：

`Result.success(data) Result.fail(message)`

### 前端代码

前端页面要按模块组织：

`api/ components/ views/ styles/`

要求：

`接口请求统一放 api 目录 通用组件放 components 页面级组件放 views 不要在页面里散落重复请求逻辑`

页面文字要清晰，不写无意义文案。

### 数据库

数据库结构以：

`docs/数据库设计.md docker/mysql/init.sql`

为准。

修改表结构时，需要同步修改：

`1. docker/mysql/init.sql 2. docs/数据库设计.md 3. 对应 Entity / Mapper / VO / DTO`

---

## 6. 文档写作规则

文档要面向“没参与开发的人也能看懂”。

写文档时优先回答：

`1. 这个东西是什么 2. 为什么需要它 3. 怎么使用 4. 常见问题是什么`

文档中命令统一使用代码块：

`docker compose up -d --build`

接口地址统一使用：

`http://localhost/api`

不要再写旧地址：

`http://localhost:8080`

除非明确说明这是开发模式下的后端直连地址。

---

## 7. Docker 与配置规则

不要提交 .env。

只提交：

`.env.example docker-compose.yml`

如果本地端口冲突，修改 .env：

`NGINX_PORT=8088 MYSQL_PORT=13306 BACKEND_PORT=18081`

不要为了自己的电脑环境直接改 docker-compose.yml。

---

## 8. 不要提交的内容

禁止提交：

`.env node_modules/ dist/ target/ .idea/ .vscode/ *.log Docker volume 数据 临时测试文件`

提交前检查：

`git status`

如果看到不确定的文件，先问负责人。

---

## 9. 合并前检查

合并到 dev 前至少确认：

`1. 项目能编译 2. 页面能打开 3. 相关接口能调用 4. 没有提交无关文件`

合并到 main 前必须确认：

`docker compose up -d --build docker compose ps powershell -ExecutionPolicy Bypass -File .\backend\scripts\api-smoke-test.ps1`

看到：

`Smoke test passed.`

才允许合并到 main。

---

## 10. 冲突处理规则

如果出现 Git 冲突：

`1. 不要随便删除别人代码 2. 先看冲突文件属于谁的模块 3. 和对应同学确认后再处理 4. 处理完必须重新运行相关功能`

冲突文件里会出现：

`<<<<<<< HEAD 当前分支内容 ======= 别人分支内容 >>>>>>> xxx`

处理后要删除这些标记。

---

## 11. 负责人职责

负责人主要负责：

`1. 维护 main 和 dev 分支 2. 审核 PR 3. 确认 Docker 部署是否通过 4. 维护 README 和部署文档 5. 控制功能范围，避免大家乱改`

负责人合并 dev 到 main 前，需要执行：

`git checkout dev git pull origin dev docker compose up -d --build powershell -ExecutionPolicy Bypass -File .\backend\scripts\api-smoke-test.ps1`

通过后：

`git checkout main git pull origin main git merge dev git push origin main`

---

## 12. 小组协作约定

建议每个人固定负责一个模块：

`前端页面 用户模块 资料模块 问答模块 后台管理 部署与文档 测试与演示`

每个人只改自己负责的模块。需要改公共代码时，先在群里说明。

公共代码包括：

`docker-compose.yml .env.example Result GlobalExceptionHandler MockTokenUtils 数据库 init.sql 公共前端请求封装`

这些文件改动影响大，不能随便改。
