# 课程资料共享平台后续完善计划

本文档用于记录项目当前设计中的不足、后续可优化方向，以及学习软件工程时需要重点理解的技术点。后续阶段可以按照本文档逐项推进实现。

## 一、总体原则

后续完善项目时，建议遵循以下原则：

1. 先保证核心业务闭环稳定，再做复杂扩展。
2. 每次只推进一个明确模块，避免同时改动过多。
3. 新增能力前先明确业务目标、数据模型、接口设计和异常场景。
4. 重要业务逻辑要补测试，避免后续重构时破坏已有功能。
5. 能用配置解决的，不写死在代码里。
6. 能抽象成接口的外部能力，不让业务模块直接依赖具体平台。

## 二、用户与认证模块

### 当前不足

1. 当前仍使用 `mock-token-{userId}`，不适合生产环境。
2. token 没有签名，任何人知道用户 ID 都可以伪造。
3. token 没有过期时间。
4. 没有刷新 token 机制。
5. 权限模型目前只使用字符串角色，扩展性有限。
6. 异常类型主要使用 `RuntimeException`，错误码还不够规范。

### 后续完善

1. 引入 JWT 或 Spring Security。
2. 增加 token 签名、过期时间和刷新机制。
3. 增加统一认证拦截器，避免每个接口手动解析 Authorization。
4. 将角色字符串改为枚举。
5. 后续可扩展 RBAC 权限模型。
6. 增加 `BusinessException` 和 `ErrorCode` 枚举。
7. 登录失败次数限制，防止暴力破解。
8. 密码修改后让旧 token 失效。

### 学习重点

1. JWT 和 Session 的区别。
2. 认证和授权的区别。
3. RBAC 权限模型。
4. BCrypt 为什么适合保存密码。
5. 为什么前端权限控制不能替代后端权限校验。

## 三、资料模块

### 当前不足

1. 资料发布流程已跑通，但文件上传成功、资料发布失败时可能产生孤儿文件。
2. 浏览量、下载量、点赞数、收藏数目前直接更新数据库，高并发下压力较大。
3. 点赞已经有幂等逻辑，但后续还可以补取消点赞。
4. 搜索目前依赖数据库 LIKE，复杂搜索能力有限。
5. 资料没有版本管理。
6. 被 AI 拒绝的资料缺少人工复审流程。

### 后续完善

1. 增加孤儿文件清理机制。
2. 资料发布失败时尝试删除已上传文件。
3. 浏览量、下载量使用 Redis 计数，定时批量同步到 MySQL。
4. 增加取消点赞接口。
5. 增加资料版本管理。
6. 增加资料人工复审接口。
7. 增加资料草稿状态。
8. 后续可接入 Elasticsearch 做搜索。
9. 增加资料推荐逻辑。

### 学习重点

1. 内容状态和上下架状态为什么要分开。
2. 多对多标签关系为什么需要中间表。
3. 幂等设计是什么。
4. 缓存一致性如何处理。
5. 为什么列表接口必须分页。

## 四、文件上传与 MinIO 模块

### 当前不足

1. 当前主要按文件扩展名限制类型。
2. 缺少明确的文件大小限制配置。
3. 文件上传成功但未绑定业务记录时，会产生孤儿文件。
4. 当前 fileUrl 偏公开访问，权限控制较弱。
5. 没有文件 MD5，无法做秒传和完整性校验。
6. 没有分片上传。
7. 没有病毒扫描或安全扫描。

### 后续完善

1. 在 Spring Boot 和 Nginx 中统一配置上传大小限制。
2. 同时校验扩展名、MIME Type 和文件头 magic number。
3. 增加文件 MD5。
4. 增加孤儿文件定时清理任务。
5. 下载时使用 MinIO 临时签名 URL。
6. 增加文件删除接口和资料下架后的对象清理。
7. 大文件场景支持分片上传。
8. 后续可接入真实云 OSS。

### 学习重点

1. 为什么文件本体不适合直接存 MySQL。
2. bucket、fileKey、fileUrl 的区别。
3. 为什么 originalFilename 不可信。
4. 对象存储和数据库如何配合。
5. 文件上传有哪些安全风险。

## 五、AI 审核模块

### 当前不足

1. 当前是 Mock 审核，只能命中简单关键词。
2. 审核结果主要是 `PASS` 和 `REJECT`，缺少 `RISK` 人工复审状态。
3. AI 服务失败时的降级策略还不完善。
4. 审核记录中可以补充模型名称、耗时、原始响应。
5. 目前主要审核文本，没有解析 PDF/DOC 文件正文。
6. 不同内容类型还没有差异化审核规则。

### 后续完善

1. 增加 `RISK` 状态。
2. 增加人工复审流程。
3. 支持通过配置选择审核提供方：mock / real。
4. 新增真实 AI 审核实现类。
5. 增加审核超时、重试、降级策略。
6. 审核失败时建议进入 `PENDING`，等待人工处理。
7. 记录模型名称、请求耗时、原始响应摘要。
8. 对长文本做截断或摘要。
9. 对文件内容做文本提取后审核。
10. 增加审核结果统计。

### 学习重点

1. 为什么要抽象 `AiAuditService`。
2. 策略模式如何用于第三方服务切换。
3. AI 审核结果和业务审核状态有什么区别。
4. AI 审核和举报治理如何配合。
5. 第三方服务失败时如何设计降级策略。
