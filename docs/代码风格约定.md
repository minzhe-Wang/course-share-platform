# 课程资料共享与互助问答平台代码风格约定

> 适用范围：本项目后端 Spring Boot 代码、接口设计、数据库访问、Git 提交与团队协作。  
> 目标：让代码稳定、清晰、易读、易维护，避免“能跑但没人敢改”的代码。

---

## 1. 总体原则

### 1.1 代码不是写给机器看的，是写给团队看的

本项目代码必须满足：

```text
能运行
能看懂
能排查
能扩展
能交接
```

禁止只追求“功能能跑”，忽视命名、分层、异常处理和接口规范。

---

### 1.2 所有人遵守同一套风格

项目中不要出现多种风格混用，例如：

```text
有人用 Result
有人直接返回 String
有人用 user_id
有人用 userId
有人把业务写 Controller
有人把业务写 Service
```

统一风格比个人习惯更重要。

---

### 1.3 优先清晰，不追求炫技

不使用复杂难懂的写法来展示技术感。

推荐：

```java
if (user == null) {
    throw new RuntimeException("用户不存在");
}
```

不推荐：

```java
Optional.ofNullable(user)
        .orElseThrow(() -> new RuntimeException("用户不存在"));
```

后者不是错，但当前项目更强调团队可读性。

---

## 2. 后端包结构约定

后端统一根包：

```text
com.example.backend
```

所有业务代码必须放在该包或其子包下，避免 Spring Boot 扫描不到。

推荐结构：

```text
com.example.backend
├── BackendApplication.java
├── common
├── config
├── controller
├── dto
├── entity
├── exception
├── mapper
├── service
│   └── impl
├── utils
└── vo
```

### 2.1 各包职责

| 包名           | 职责                          |
| ------------ | --------------------------- |
| common       | 通用返回对象、常量、枚举                |
| config       | 配置类，如 MinIO、跨域、MyBatis 配置   |
| controller   | 接收 HTTP 请求，做参数校验，调用 Service |
| dto          | 前端请求参数对象                    |
| entity       | 数据库表对应实体                    |
| exception    | 全局异常处理、自定义异常                |
| mapper       | MyBatis 数据访问接口              |
| service      | 业务接口                        |
| service.impl | 业务实现                        |
| utils        | 工具类                         |
| vo           | 返回给前端的展示对象                  |

## 3. 分层调用约定

### 3.1 调用方向

必须遵守：

```text
Controller → Service → Mapper → Database
```

禁止跨层调用：

```text
Controller 不能直接调用 Mapper
Service 不能返回数据库 ResultSet
Mapper 不能写业务判断
```

### 3.2 Controller 职责

Controller 只做三件事：

```text
1. 接收请求参数
2. 调用 Service
3. 返回 Result
```

推荐：

```java
@PostMapping("/login")
public Result<LoginVO> login(@RequestBody @Valid LoginDTO loginDTO) {
    return Result.success(userService.login(loginDTO));
}
```

不推荐：

```java
@PostMapping("/login")
public Result<LoginVO> login(@RequestBody LoginDTO loginDTO) {
    SysUser user = userMapper.findByUsername(loginDTO.getUsername());
    if (user == null) {
        return Result.fail("用户不存在");
    }
    // 大量业务逻辑...
}
```

### 3.3 Service 职责

Service 负责业务规则，例如：

```text
用户名是否已存在
密码是否正确
用户是否被禁用
资料是否审核通过
是否已经点赞
是否有权限处理举报
```

### 3.4 Mapper 职责

Mapper 只负责数据库操作：

```text
select
insert
update
delete
```

Mapper 中不要写复杂业务判断。

## 5. Entity / DTO / VO 使用规范

### 5.1 Entity

Entity 对应数据库表，字段尽量与表字段一一对应。

推荐：

```java
@Data
@Builder
@NoArgsConstructor
@AllArgsConstructor
public class SysUser {

    private Long id;
    private String username;
    private String password;
    private String nickname;
    private String role;
    private Integer status;
    private LocalDateTime createTime;
    private LocalDateTime updateTime;
}
```

Entity 不写业务逻辑。

---

### 5.2 DTO

DTO 用于接收前端请求参数。

示例：

```java
@Data
public class LoginDTO {

    @NotBlank(message = "用户名不能为空")
    private String username;

    @NotBlank(message = "密码不能为空")
    private String password;
}
```

DTO 中应使用参数校验注解：

```text
@NotBlank
@NotNull
@Size
@Min
@Max
```

---

### 5.3 VO

VO 用于返回给前端，不直接返回 Entity。

推荐：

```java
@Data
@Builder
public class LoginVO {

    private Long userId;
    private String username;
    private String nickname;
    private String role;
    private String token;
}
```

禁止直接把 `SysUser` 返回给前端，因为其中包含：

```text
password
status
createTime
updateTime
```

尤其不能返回 `password`。

---

### 5.4 DTO / Entity / VO 的区别

```text
DTO：前端传给后端
Entity：后端对应数据库
VO：后端返回给前端
```

示例：

```text
LoginDTO：username + password
SysUser：id + username + password + role + status
LoginVO：userId + username + nickname + role + token
```

## 6. Lombok 使用规范

### 6.1 推荐使用

Entity：

```java
@Data
@Builder
@NoArgsConstructor
@AllArgsConstructor
public class SysUser {
}
```

DTO：

```java
@Data
public class LoginDTO {
}
```

VO：

```java
@Data
@Builder
public class LoginVO {
}
```

### 6.2 不推荐使用

Controller、Service、Mapper 不使用 `@Data`。

不推荐在团队项目中滥用：

```java
@Accessors(chain = true)
```

理由：链式 setter 容易让不熟悉 Lombok 的组员困惑。

### 6.3 创建对象推荐 Builder

推荐：

```java
SysUser user = SysUser.builder()
        .username(username)
        .password(encodedPassword)
        .nickname(nickname)
        .role("STUDENT")
        .status(1)
        .build();
```

不推荐：

```java
SysUser user = new SysUser();
user.setUsername(username);
user.setPassword(encodedPassword);
user.setNickname(nickname);
user.setRole("STUDENT");
user.setStatus(1);
```

不是不能用，而是 Builder 更清晰。

## 7. 统一返回规范

所有接口必须返回 `Result<T>`。

成功：

```json
{
  "code": 200,
  "message": "success",
  "data": {}
}
```

失败：

```json
{
  "code": 500,
  "message": "用户名已存在",
  "data": null
}
```

### 7.1 Result 示例

```java
public class Result<T> {

    private Integer code;
    private String message;
    private T data;

    public static <T> Result<T> success() {
        return new Result<>(200, "success", null);
    }

    public static <T> Result<T> success(T data) {
        return new Result<>(200, "success", data);
    }

    public static <T> Result<T> fail(String message) {
        return new Result<>(500, message, null);
    }

    public static <T> Result<T> fail(Integer code, String message) {
        return new Result<>(code, message, null);
    }
}
```

### 7.2 Controller 返回约定

推荐：

```java
return Result.success(userService.login(loginDTO));
```

不推荐：

```java
return loginVO;
```

不推荐：

```java
return "success";
```

## 8. 异常处理规范

### 8.1 业务失败使用异常抛出

推荐：

```java
if (existUser != null) {
    throw new RuntimeException("用户名已存在");
}
```

Controller 不要到处写失败返回。

### 8.2 全局异常统一处理

使用：

```java
@RestControllerAdvice
public class GlobalExceptionHandler {
}
```

统一返回：

```json
{
  "code": 500,
  "message": "用户名已存在",
  "data": null
}
```

### 8.3 异常信息必须给人看

推荐：

```java
throw new RuntimeException("用户名已存在");
throw new RuntimeException("密码错误");
throw new RuntimeException("资料不存在");
throw new RuntimeException("无权限处理该举报");
```

不推荐：

```java
throw new RuntimeException("error");
throw new RuntimeException("fail");
throw new RuntimeException("bad request");
```

## 10. 参数校验规范

所有请求 DTO 中应写参数校验。

示例：

```java
@Data
public class RegisterDTO {

    @NotBlank(message = "用户名不能为空")
    @Size(min = 3, max = 30, message = "用户名长度必须在3到30位之间")
    private String username;

    @NotBlank(message = "密码不能为空")
    @Size(min = 6, max = 30, message = "密码长度必须在6到30位之间")
    private String password;
}
```

Controller 中必须加 `@Valid`：

```java
@PostMapping("/register")
public Result<Void> register(@RequestBody @Valid RegisterDTO registerDTO) {
    userService.register(registerDTO);
    return Result.success();
}
```

## 11. 数据库访问规范

### 11.1 MyBatis Mapper 命名

推荐：

```java
findById()
findByUsername()
insert()
updateStatus()
deleteById()
```

不推荐：

```java
select1()
getInfo()
updateData()
doSql()
```

### 11.2 SQL 字段不要写 `SELECT *`

推荐：

```java
@Select("""
        SELECT id, username, password, nickname, role, status, create_time, update_time
        FROM sys_user
        WHERE username = #{username}
        LIMIT 1
        """)
SysUser findByUsername(String username);
```

不推荐：

```java
@Select("SELECT * FROM sys_user WHERE username = #{username}")
SysUser findByUsername(String username);
```

原因：

```text
字段清晰
减少无用字段
方便维护
避免表结构变化导致不可控问题
```

### 11.3 修改多个表时使用事务

例如资料发布：

```text
insert material
insert material_tag
insert ai_audit_record
update material.audit_status
```

必须加：

```java
@Transactional
```

示例：

```java
@Transactional
public MaterialCreateVO createMaterial(MaterialCreateDTO dto) {
    // 多表操作
}
```

user.setRole(RoleConstants.STUDENT);

## 13. 注释规范

### 13.1 不写废话注释

不推荐：

```java
// 获取用户名
String username = loginDTO.getUsername();
```

代码本身已经说明了。

### 13.2 需要写注释的场景

```text
复杂业务规则
重要流程
容易踩坑的代码
外部服务调用
特殊 SQL
```

推荐：

```java
// 第一版暂时使用 mock token，后续替换为 JWT
String token = "mock-token-" + user.getId();
```

### 13.3 方法注释不强制

简单 CRUD 方法不用每个都写 JavaDoc。

但复杂业务方法建议写：

```java
/**
 * 发布资料：
 * 1. 保存资料元信息
 * 2. 绑定标签
 * 3. 调用 AI 审核
 * 4. 写入 AI 审核记录
 * 5. 更新资料审核状态
 */
@Transactional
public MaterialCreateVO createMaterial(MaterialCreateDTO dto) {
}
```

## 14. Service 代码风格

### 14.1 推荐早返回，减少嵌套

推荐：

```java
if (user == null) {
    throw new RuntimeException("用户不存在");
}

if (user.getStatus() == 0) {
    throw new RuntimeException("用户已被禁用");
}
```

不推荐：

```java
if (user != null) {
    if (user.getStatus() != 0) {
        // 业务逻辑
    }
}
```

### 14.2 一个方法不要太长

建议一个业务方法控制在 80 行以内。

如果超过 80 行，考虑拆分私有方法：

```java
private void checkUsernameAvailable(String username) {
}

private String buildMockToken(Long userId) {
}
```

## 23. 示例：用户登录标准写法

Controller：

```java
@PostMapping("/login")
public Result<LoginVO> login(@RequestBody @Valid LoginDTO loginDTO) {
    return Result.success(userService.login(loginDTO));
}
```

Service：

```java
@Override
public LoginVO login(LoginDTO loginDTO) {
    String username = loginDTO.getUsername().trim();

    SysUser user = userMapper.findByUsername(username);
    if (user == null) {
        throw new RuntimeException("用户不存在");
    }

    if (user.getStatus() == null || user.getStatus() == 0) {
        throw new RuntimeException("用户已被禁用");
    }

    boolean passwordMatch = passwordEncoder.matches(loginDTO.getPassword(), user.getPassword());
    if (!passwordMatch) {
        throw new RuntimeException("密码错误");
    }

    return LoginVO.builder()
            .userId(user.getId())
            .username(user.getUsername())
            .nickname(user.getNickname())
            .role(user.getRole())
            .token("mock-token-" + user.getId())
            .build();
}
```

Mapper：

```java
@Select("""
        SELECT id, username, password, nickname, role, phone, email, avatar, status, create_time, update_time
        FROM sys_user
        WHERE username = #{username}
        LIMIT 1
        """)
SysUser findByUsername(String username);
```

## 25. 总结

本项目代码风格核心要求：

```text
分层清晰
命名准确
参数校验
统一返回
统一异常
DTO / Entity / VO 分离
Controller 简洁
Service 表达业务
Mapper 专注 SQL
Apifox 与代码同步
Git 提交清晰
```

最终目标不是“代码越短越好”，而是：

```text
别人一看就懂
出错容易排查
后续容易扩展
答辩能够讲清楚
```
