# 课程资料共享与互助问答平台设计文档（最新版）

## 一、项目概述

### 1. 项目名称

**基于 AI 内容审核的课程资料共享与互助问答平台**

---

### 2. 项目背景

在高校学习过程中，学生经常需要获取课程课件、试卷、实验指导、复习资料、作业参考等学习资源，同时也会在课程学习中遇到各种问题。现有资料通常分散在班级群、网盘、个人电脑或不同课程平台中，存在资料查找困难、版本混乱、资源质量参差不齐、交流渠道不统一等问题。

因此，本项目拟设计并实现一个面向高校学生的课程资料共享与互助问答平台。平台支持学生上传、搜索、下载、收藏、点赞课程资料，并提供课程问答、回答、楼中楼回复、点赞、举报等功能。

根据最新设计调整，系统引入 **AI 内容审核机制**。资料简介、问题、回答和回复等用户提交内容优先由 AI 自动审核，审核通过后公开展示；审核不通过则驳回。审核员不再负责逐条人工审核普通内容，而是主要负责处理用户举报和争议内容。系统同时引入 **Docker 容器化部署方案**，通过 Docker Compose 统一编排后端、数据库、缓存、对象存储等服务，降低小组成员本地环境配置成本。

---

### 3. 项目目标

本项目目标如下：

1. 建立统一的课程资料共享平台，提高学生获取学习资源的效率。
2. 支持课程资料上传、搜索、下载、收藏、点赞和热门排序。
3. 建立课程问答区，支持学生发布问题、回答问题、楼中楼回复和点赞互动。
4. 引入 AI 内容审核机制，提高资料和问答内容审核效率。
5. 引入举报与人工兜底机制，保障平台内容质量。
6. 提供后台管理能力，实现用户、课程分类、标签、内容和举报的统一管理。
7. 采用前后端分离、单体后端、MySQL + Redis + 对象存储的架构，保证系统结构清晰、适合课程项目实现。
8. 采用 Docker Compose 进行项目部署，降低环境配置成本，提高团队协作效率。

---

## 二、系统定位

系统定位为：

```text
课程资料共享 + 课程互助问答 + AI 内容审核 + 举报治理平台
```

平台主要解决三个问题：

```text
1. 学习资料分散，难以统一查找。
2. 课程问题缺少集中讨论和沉淀。
3. 用户生成内容需要审核与治理。
```

系统主要面向三类用户：

```text
2. 审核员 REVIEWER
3. 管理员 ADMIN
```

## 四、技术架构

### 1. 技术选型

```text
前端：Vue3 + Vite
后端：Spring Boot 3 + Java 17
数据库：MySQL 8
缓存：Redis
文件存储：OSS / MinIO
反向代理：Nginx
部署方式：Docker + Docker Compose
AI 审核：AI 审核服务 / 大模型接口 / 模拟审核服务
```

说明：

- 课程项目开发阶段可使用 MinIO 模拟 OSS。
- AI 审核可先设计接口并实现模拟审核，后续可以对接真实大模型 API。
- Docker Compose 用于一键启动 MySQL、Redis、MinIO、后端服务和 Nginx。

---

### 2. 总体架构

```text
用户浏览器
   ↓
Nginx 容器
   ↓
Vue 前端页面 / Spring Boot API
   ↓
Spring Boot 后端容器
   ├── MySQL 容器：业务数据
   ├── Redis 容器：热点缓存
   ├── MinIO / OSS：资料文件存储
   └── AI 审核服务：内容自动审核
```

---

### 3. 部署架构

```text
Docker Compose
├── frontend 容器：前端静态资源

├── backend 容器：Spring Boot 后端
├── mysql 容器：MySQL 8
├── redis 容器：Redis

```

```text
问答相关：
- 发布问题
- 回答问题
- 回复回答

- 查看我的回答
```

---

### 2. 审核员 REVIEWER

最新版设计中，审核员不再处理普通资料和问答内容的逐条审核，而是主要负责处理举报和争议内容。

审核员拥有以下权限：

```text
举报处理：
- 查看举报列表
- 查看被举报内容
- 判断举报是否成立
- 下架违规内容
- 驳回无效举报
- 填写处理结果
```

审核员不具备以下权限：

```text
- 上传资料
- 收藏资料
- 点赞资料
- 发布问题
- 回答问题
- 回复回答
- 点赞问答内容
```

说明：

```text
资料、问题、回答、回复的普通审核由 AI 自动完成；
审核员负责人工兜底，处理 AI 漏审、误审或用户举报内容。
```

---

### 3. 管理员 ADMIN

管理员负责系统整体管理和平台治理。

管理员拥有以下权限：

```text
用户管理：
- 查看用户列表
- 启用/禁用用户

基础数据管理：
- 课程分类管理
- 标签管理

内容管理：
- 资料管理
- 问题管理
- 回答管理
- 回复管理

举报处理：
- 查看举报记录
- 处理举报
- 下架违规内容
- 驳回无效举报

系统管理：
- 查看 AI 审核记录
- 查看系统基础统计
```

管理员不作为普通用户参与问答互动，不发布问题、不回答、不点赞。

---

### 4. 权限设计说明

本系统采用简化 RBAC 设计。由于系统角色固定为：

```text
STUDENT / REVIEWER / ADMIN
```

权限数量有限，因此首版不额外设计 `role_permission` 权限表，而是在后端通过 `sys_user.role` 字段进行权限判断。

答辩表述：

> 本系统采用基于角色的权限控制。学生负责内容发布和互动，审核员负责举报处理和人工兜底，管理员负责系统管理和平台治理。由于角色和权限较固定，因此首版不设计复杂权限表，而是通过用户表中的 role 字段进行权限判断。

## 八、课程分类与标签模块

### 1. 课程分类

课程分类用于表示资料或问题所属的课程维度。

示例：

```text
操作系统
计算机网络
数据结构
软件工程
数据库原理
```

课程分类是强约束。每份资料、每个问题都必须绑定一个课程分类。

---

### 2. 标签

标签用于描述资料属性，是弱约束，一个资料可以有多个标签。

标签类型包括：

```text
GRADE：年级标签
TYPE：资料类型标签
SCENE：使用场景标签
```

示例：

```text
GRADE：
大一 / 大二 / 大三 / 大四

TYPE：
试卷 / 课件 / 笔记 / 实验指导 / 复习资料 / 作业答案 / 项目代码

SCENE：
期末复习 / 考研 / 课程设计 / 平时作业
```

---

### 3. 分类与标签区别

```text
分类 = 资料属于哪门课
标签 = 资料有什么属性
```

例如：

```text
资料：操作系统期末试卷.pdf

课程分类：操作系统
标签：试卷 / 大二 / 期末复习
文件类型：PDF
```

注意：PDF、DOCX、ZIP 是文件类型，存储在 `material.file_type` 字段中，不作为标签。

## 九、课程资料模块

### 1. 资料上传

学生可以上传课程资料。支持文件类型：

```text
PDF
DOC
DOCX
ZIP
```

资料文件本体存储到 OSS / MinIO，数据库中只保存文件访问地址、对象 Key、原始文件名、文件类型和文件大小。

上传资料时，学生需要填写：

```text
- 资料标题
- 资料简介
- 课程分类
- 标签
- 文件
```

上传后资料默认进入：

```text
PENDING 待审核
```

随后系统调用 AI 审核服务进行自动审核。

---

### 2. 资料 AI 审核

资料上传后，系统对资料标题、简介、原始文件名等文本信息进行 AI 审核。

流程：

```text
学生上传资料
→ 后端保存文件到 OSS / MinIO
→ 数据库保存资料元信息
→ material.audit_status = PENDING
→ 调用 AI 审核服务
→ 保存 AI 审核记录
→ 通过：material.audit_status = APPROVED
→ 不通过：material.audit_status = REJECTED
```

说明：

```text
课程首版可以先审核标题、简介、文件名；
如需进一步扩展，可对 PDF / DOCX 文本内容进行解析后送审。
```

---

### 3. 资料列表

系统展示已审核通过且状态正常的资料列表，支持：

```text
- 分页查询
- 按课程分类筛选
- 按标签筛选
- 按关键词搜索
- 按点赞数排序
- 按收藏数排序
- 按上传时间排序
```

查询条件：

```text
audit_status = APPROVED
status = 1
```

---

### 4. 资料详情

资料详情展示：

```text
- 资料标题
- 资料简介
- 所属课程分类
- 标签
- 文件类型
- 文件大小
- 上传者
- 上传时间
- 浏览数
- 下载数
- 点赞数
- 收藏数
```

---

### 5. 资料下载

用户可以下载已审核通过的资料。系统记录下载行为，包括用户、资料、下载时间和 IP。

流程：

```text
用户进入资料详情页
→ 点击下载
→ 后端校验资料是否审核通过、状态是否正常
→ 记录下载行为
→ 更新 download_count
→ 返回 OSS / MinIO 文件下载地址
→ 用户下载资料
```

---

### 6. 资料收藏

学生可以收藏资料。系统通过收藏记录表保存用户与资料之间的收藏关系，并在资料表中维护 `favorite_count`，用于排序和推荐。

---

### 7. 资料点赞

学生可以对资料点赞。系统通过点赞记录表防止重复点赞，并在资料表中维护 `like_count`，用于热门资料排序。

---

### 8. 资料举报

用户可以举报违规资料。举报内容进入举报管理流程，由审核员或管理员处理。

## 十、互助问答模块

### 1. 问答结构

问答区采用最多两层结构：

```text
问题 question
  └── 回答 answer
        └── 回复 answer_reply
```

系统支持“回复某条回复”，但不继续增加数据库层级。所有楼中楼内容都平铺存储在 `answer_reply` 表中，通过 `reply_to_user_id` 表示回复对象。

示例：

```text
问题：进程和线程有什么区别？

回答：进程是资源分配单位，线程是调度单位。

回复A：这里的资源具体指什么？
回复B 回复A：主要是内存空间、文件句柄等。
回复C 回复B：那线程共享这些资源吗？
```

展示上属于同一个回答下的回复列表，不做无限嵌套。

---

### 2. 发布问题

学生可以发布课程相关问题。问题必须绑定课程分类，发布后默认进入待审核状态，并由 AI 自动审核。

流程：

```text
学生发布问题
→ question.audit_status = PENDING
→ 调用 AI 审核
→ 通过：APPROVED，公开展示
→ 不通过：REJECTED，不公开展示
```

---

### 3. 回答问题

学生可以对已审核通过的问题进行回答。回答提交后默认进入待审核状态，并由 AI 自动审核。

---

### 4. 回复回答

学生可以在回答下进行回复，也可以回复其他用户的回复。回复提交后默认进入待审核状态，并由 AI 自动审核。

---

### 5. 问答点赞

学生可以对以下对象点赞：

```text
问题
回答
回复
```

系统通过 `like_record` 记录用户点赞行为，并维护对应对象的 `like_count` 字段。

---

### 6. 问答举报

用户可以举报：

```text
问题
回答
回复
```

举报后进入人工处理流程，由审核员或管理员判断是否违规。

## 十一、AI 审核模块

### 1. 模块定位

AI 审核模块用于对用户提交的资料和问答内容进行自动审核，减少人工审核压力。

审核对象包括：

```text
MATERIAL：资料
QUESTION：问题
ANSWER：回答
REPLY：回复
```

---

### 2. 审核结果

AI 审核结果包括：

```text
PASS：通过
REJECT：不通过
RISK：疑似风险
```

建议首版处理方式：

```text
PASS → 业务表 audit_status = APPROVED
REJECT → 业务表 audit_status = REJECTED
RISK → 业务表 audit_status = REJECTED 或保留 PENDING
```

为了简化课程项目，建议：

```text
PASS → APPROVED
REJECT / RISK → REJECTED
```

如果想体现“人工兜底”，也可以设计为：

```text
RISK → PENDING，进入人工复核
```

但这会增加审核员工作量。根据当前助教建议，推荐首版采用 AI 自动决定，审核员只处理举报。

---

### 3. AI 审核记录

每次 AI 审核都写入 `ai_audit_record` 表，记录：

```text
- 审核对象类型
- 审核对象 ID
- AI 审核结果
- 风险分数
- 审核原因
- 模型名称
- 送审内容摘要
- 原始返回内容
- 创建时间
```

---

### 4. AI 审核流程

```text
用户提交内容
  ↓
后端保存内容，状态设为 PENDING
  ↓
调用 AI 审核服务
  ↓
AI 返回 PASS / REJECT / RISK
  ↓
保存 ai_audit_record
  ↓
更新业务表 audit_status / audit_remark / audit_time
  ↓
通过则公开展示，不通过则隐藏
```

---

### 5. AI 审核服务实现方式

课程项目可以采用三阶段实现：

```text
第一阶段：Mock 模拟审核
- 通过关键词规则判断违规内容
- 不依赖外部 API
- 方便小组开发和演示

第二阶段：接口抽象
- 定义 AiAuditService
- 后续可以替换为真实大模型 API

第三阶段：真实 AI 接入
- 接入大模型 API 或内容安全 API
- 返回风险分、风险原因和审核结果
```

推荐首版：

```text
先实现 MockAiAuditService，保证项目可运行；
答辩时说明系统预留 AI 审核接口，可以替换真实模型服务。
```

## 十二、举报处理模块

### 1. 举报对象

举报对象包括：

```text
MATERIAL
QUESTION
ANSWER
REPLY
```

---

### 2. 举报状态

举报处理状态包括：

```text
PENDING：待处理
RESOLVED：认定违规，已处理
REJECTED：认定不违规，驳回举报
```

---

### 3. 举报处理流程

```text
用户举报资料/问题/回答/回复
→ 系统保存举报记录和内容快照
→ 审核员查看举报详情
→ 审核员判断是否违规
→ 若违规，下架对应内容
→ 若不违规，驳回举报
→ 记录处理结果
```

---

### 4. 内容治理模式

最新版内容治理模式：

```text
AI 前置审核 + 用户举报 + 审核员人工兜底
```

答辩表述：

> 本系统采用“AI 前置审核 + 用户举报 + 人工兜底”的内容治理模式。普通内容由 AI 自动审核，提高审核效率；对于 AI 漏审、误审或用户争议内容，则通过举报机制进入人工处理流程。

## 十四、核心业务流程

### 1. 资料上传与 AI 审核流程

```text
学生上传资料
→ 后端校验文件类型和文件大小
→ 文件上传至 OSS / MinIO
→ 数据库保存资料元信息
→ 资料状态设为 PENDING
→ 调用 AI 审核服务
→ 写入 AI 审核记录
→ AI 审核通过，资料状态改为 APPROVED
→ AI 审核不通过，资料状态改为 REJECTED
```

---

### 2. 资料下载流程

```text
用户进入资料详情页
→ 点击下载
→ 后端校验资料是否审核通过、状态是否正常
→ 记录下载行为
→ 更新资料下载次数
→ 返回 OSS / MinIO 文件下载地址
→ 用户下载资料
```

---

### 3. 资料点赞 / 收藏流程

```text
学生点击点赞或收藏
→ 后端校验用户角色是否为 STUDENT
→ 检查是否已经点赞/收藏
→ 新增点赞/收藏记录
→ 更新 material.like_count 或 material.favorite_count
→ 返回操作结果
```

---

### 4. 问题发布与 AI 审核流程

```text
学生发布问题
→ 绑定课程分类
→ 系统保存问题
→ 状态设为 PENDING
→ 调用 AI 审核服务
→ 写入 AI 审核记录
→ AI 审核通过后公开展示
→ AI 审核不通过则驳回
```

---

### 5. 回答与回复流程

```text
学生回答问题
→ 回答进入 PENDING
→ AI 自动审核
→ 通过后展示在问题详情页

学生回复回答或回复其他用户
→ 回复进入 PENDING
→ AI 自动审核
→ 通过后展示在回答下方
```

---

### 6. 举报处理流程

```text
用户举报资料/问题/回答/回复
→ 系统保存举报记录和内容快照
→ 审核员查看举报详情
→ 判断是否违规
→ 若违规，下架对应内容
→ 若不违规，驳回举报
→ 记录处理结果
```

## 二十二、非功能需求

### 1. 安全性

```text
- 用户密码加密存储
- 根据角色控制接口权限
- 审核员和管理员不能作为普通用户参与点赞、收藏、发布问答
- 文件上传限制类型和大小
- 防止 SQL 注入
- AI 审核机制减少违规内容发布
- 举报和人工处理机制防止违规内容扩散
```

---

### 2. 可用性

```text
- 页面结构清晰
- 支持分页、搜索、筛选和排序
- 资料列表和问答列表支持多条件查询
- 问答区控制为两层结构，避免无限嵌套影响体验
```

---

### 3. 可维护性

```text
- 后端采用 controller / service / mapper / entity 分层结构
- 数据库命名统一
- 审核字段统一
- 举报对象和点赞对象统一使用 target_type + target_id
- AI 审核记录单独成表
- 文件存储与业务数据分离
- Docker Compose 统一开发和部署环境
```

---

### 4. 性能要求

```text
- 列表类接口必须分页
- 热门资料和热门问答可使用 Redis 缓存
- 资料文件不直接经过数据库存储
- 点赞数、收藏数、回答数、回复数采用冗余计数字段，避免频繁 count 查询
```

## 二十五、答辩亮点总结

```text
1. 前后端分离架构
2. Docker Compose 一键部署
3. MySQL + Redis + 对象存储
4. AI 自动内容审核
5. 用户举报 + 审核员人工兜底
6. 资料共享 + 问答互动一体化
7. 统一点赞表 like_record
8. 统一举报表 report
9. 统一 AI 审核记录表 ai_audit_record
10. 问答楼中楼最多两层，兼顾功能和复杂度
```

答辩可用总结：

> 我们的系统不是简单的资料上传下载平台，而是一个集课程资料共享、互助问答、AI 内容审核、举报治理和容器化部署于一体的课程学习平台。系统通过 Docker Compose 降低部署成本，通过 AI 审核减少人工审核压力，并通过举报机制实现人工兜底，整体结构清晰，适合课程项目落地实现。
