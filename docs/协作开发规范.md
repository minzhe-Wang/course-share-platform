# 协作开发规范

本文档用于约定小组成员如何使用 Git 分支、如何提交代码、如何发起 Pull Request，以及负责人如何合并和发布稳定版本。

## 1. 分支模型

项目使用三类分支：

```text
main
dev
feature/xxx
```

| 分支 | 作用 | 谁来维护 |
| --- | --- | --- |
| `main` | 稳定版本，只放可以运行、可以演示、可以 Docker 部署的代码 | 项目负责人 |
| `dev` | 日常集成分支，所有功能先合并到这里 | 项目负责人 |
| `feature/xxx` | 每个人自己的功能开发分支 | 对应开发同学 |

禁止直接在 `main` 上开发。

## 2. 分支使用流程

负责人先创建并推送 `dev`：

```powershell
git checkout main
git pull origin main
git checkout -b dev
git push origin dev
```

同学开发新功能时，从 `dev` 拉分支：

```powershell
git checkout dev
git pull origin dev
git checkout -b feature/功能名
```

示例：

```powershell
git checkout -b feature/frontend-login
git checkout -b feature/material-upload
git checkout -b feature/question-answer
git checkout -b feature/admin-review
```

开发完成后：

```powershell
git add -A
git commit -m "feat: 实现资料发布页面"
git push origin feature/material-upload
```

然后在 GitHub 发起 Pull Request：

```text
feature/xxx -> dev
```

负责人审核通过后合并到 `dev`。

## 3. 分支命名规则

| 类型 | 用途 | 示例 |
| --- | --- | --- |
| `feature/xxx` | 新功能 | `feature/material-upload` |
| `fix/xxx` | 修复问题 | `fix/chinese-encoding` |
| `docs/xxx` | 文档修改 | `docs/deployment-guide` |
| `refactor/xxx` | 重构 | `refactor/user-service` |

命名要求：

- 使用英文小写。
- 单词之间用 `-`。
- 名字要能看出做什么。

## 4. 提交信息规则

提交信息格式：

```text
类型: 具体说明
```

| 类型 | 说明 |
| --- | --- |
| `feat` | 新增功能 |
| `fix` | 修复问题 |
| `docs` | 修改文档 |
| `style` | 格式调整，不影响功能 |
| `refactor` | 代码重构 |
| `test` | 添加或修改测试 |
| `chore` | 构建、配置、依赖等杂项 |

推荐：

```text
feat: 实现资料发布接口
feat: 实现问答列表页面
fix: 修复 Docker 端口冲突
fix: 修复中文乱码问题
docs: 补充开发模式说明
test: 增加资料模块接口测试
refactor: 调整用户模块 Service 分层
```

不推荐：

```text
update
修改
111
最终版
改了一点
```

## 5. Pull Request 规则

每个 PR 需要说明：

```text
1. 做了什么
2. 改了哪些模块
3. 如何验证
4. 是否有遗留问题
```

PR 模板：

```markdown
## 本次改动

- 

## 涉及模块

- 

## 验证方式

- [ ] 本地后端启动成功
- [ ] 本地前端启动成功
- [ ] Docker 部署通过
- [ ] 冒烟测试通过
- [ ] 浏览器页面验证通过

## 备注

-
```

文档类 PR 可以简化：

```markdown
## 本次改动

- 补充 Docker 部署教程
- 补充开发模式说明

## 验证方式

- [x] 检查文档链接可读
```

## 6. 合并规则

合并到 `dev` 前至少确认：

```text
1. 代码能编译
2. 页面能打开
3. 相关接口能调用
4. 没有提交无关文件
5. 没有明显影响其他模块
```

合并到 `main` 前必须执行：

```powershell
docker compose up -d --build
docker compose ps
powershell -ExecutionPolicy Bypass -File .\backend\scripts\api-smoke-test.ps1
```

看到：

```text
Smoke test passed.
```

才允许合并到 `main`。

## 7. 负责人合并 dev 到 main

负责人操作：

```powershell
git checkout dev
git pull origin dev
docker compose up -d --build
powershell -ExecutionPolicy Bypass -File .\backend\scripts\api-smoke-test.ps1
```

验证通过后：

```powershell
git checkout main
git pull origin main
git merge dev
git push origin main
```

`main` 必须始终保持可运行、可演示、可部署。

## 8. 不要提交的内容

禁止提交：

```text
.env
node_modules/
dist/
target/
.idea/
.vscode/
*.log
Docker volume 数据
临时测试文件
```

提交前必须看：

```powershell
git status
```

如果看到不确定的文件，先问负责人。

## 9. 公共文件修改规则

以下文件影响范围较大，修改前要先说明：

```text
docker-compose.yml
.env.example
docker/mysql/init.sql
nginx/nginx.conf
backend/src/main/java/com/example/backend/common/Result.java
backend/src/main/java/com/example/backend/exception/GlobalExceptionHandler.java
backend/src/main/java/com/example/backend/utils/MockTokenUtils.java
frontend/src/api/
```

修改公共文件后，必须说明原因，并重新验证相关功能。

## 10. 冲突处理规则

如果出现 Git 冲突：

```text
1. 不要随便删除别人代码
2. 先判断冲突文件属于哪个模块
3. 和对应同学确认后再处理
4. 处理完必须重新运行相关功能
```

冲突标记示例：

```text
<<<<<<< HEAD
当前分支内容
=======
别人分支内容
>>>>>>> feature/xxx
```

处理完后要删除这些标记。

## 11. 最简协作规则

如果只记住一版，就记住这些：

```text
不要直接改 main。
所有人从 dev 拉 feature 分支。
做完后 push 自己的 feature 分支。
通过 PR 合并到 dev。
dev 测试通过后，负责人合并到 main。
提交前一定看 git status。
.env、node_modules、target、dist 不提交。
main 必须保持能 Docker 跑通。
```
